Every project here ships with invariant tests and a benchmark. The invariant defines what must always be true. The benchmark proves it holds under load. Anyone can generate code that compiles. The harder question is: what property did you choose to guarantee, and how did you verify it?

Pre-Trade Analytics Dashboard

GitHub Live demo
The problem

Options pricing libraries are easy to write and hard to trust. Most implementations get the happy path right but silently produce wrong Greeks under edge conditions -- deep out-of-the-money, near-zero expiry, extreme volatility. The errors compound when you build scenario analysis or P&L attribution on top.

The decision

Instead of testing outputs against a textbook table (which only covers a few cases), I verified put-call parity as a structural invariant -- it must hold to 1e-10 for every combination of inputs. If parity breaks, the entire pricing model is wrong. This catches classes of bugs that example-based tests miss. I also built the engine to run client-side so the demo works without a backend -- the math is the same, the deployment is simpler.

What I'd change

The current model assumes constant volatility. Real options markets have a volatility surface. Adding surface interpolation and re-running the parity check across the surface would be the next step -- same verification principle, harder problem.

Delivery Order Router

GitHub
The problem

In a delivery system, a dropped order is worse than a slow one. Naive routing assigns orders without checking dasher capacity or verifying uniqueness -- under concurrent load, orders get double-assigned or fall through entirely. The failure mode is silent: no error, just a missing delivery.

The decision

I defined two invariants before writing any routing logic: every order must be assigned exactly once, and no dasher can exceed capacity. The 6 test suite runs those checks after every assignment round. The 1000-request benchmark isn't about showing speed -- it's about proving zero violations at scale. 1,211 req/s is a side effect of getting the data model right.

What I'd change

The current router is greedy (assign to least-loaded dasher). A cost-minimizing approach that accounts for dasher location and delivery distance would be more realistic. The invariants stay the same -- the routing strategy changes, the correctness guarantee doesn't.

Distributed Generation Portfolio Monitor

GitHub
The problem

Anomaly detection on a mixed-scale fleet is a harder problem than it looks. A 5kW rooftop panel and a 500kW commercial array have completely different output profiles. A single threshold that works for the large asset is blind to faults on the small one. Naive monitoring on this fleet catches 0 out of 3 injected faults.

The decision

I segmented assets by capacity class before computing anomaly thresholds. Each segment gets its own baseline derived from its own distribution. The benchmark injects 3 known faults across the mixed fleet and compares segmented vs. naive detection. Segmented catches all 3 with a false positive rate of 0.003. Naive catches zero -- it's not slightly worse, it's completely blind.

What I'd change

The current segmentation is manual (small vs. large). A clustering approach that discovers segments from the data would generalize better to fleets with three or more asset classes. The FPR benchmark should also run on seasonal data -- summer and winter profiles differ enough to change the thresholds.
